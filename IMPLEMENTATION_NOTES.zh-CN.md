# 实现说明

## 这次改动的目标

这个项目现在分成两层：

- 一层是面向通用 Codex CLI 场景的 macOS 通知能力
- 另一层是面向 Ghostty 多标签场景的可选增强包装器

基础层解决的是通用问题：

- 当 Codex 在后台完成任务时，用户应该能及时收到提醒
- 当 Codex 需要命令批准或额外输入时，用户也应该被提醒，而不是错过交互时机

Ghostty 扩展层解决的是基础方案覆盖不到的一个更细的问题：

- 即使 Ghostty 仍然是前台应用，只要你已经切到了别的 tab，而审批请求还挂着，也应该发桌面通知

## 实现方法

### 基础层：官方 hook 路径处理任务完成

对于任务完成事件，最干净的方式是用官方 `notify` 配置。Codex 可以在一个 turn 完成后调用本地程序，并把 JSON payload 传给它。

在这个项目里，hook 指向的是 `codex-notify.mjs`。

它的职责刻意保持很小：

1. 接收原始 payload
2. 安全解析 JSON
3. 提取简洁的标题和正文
4. 可选追加一条机器可读的本地日志
5. 调用 `osascript` 发出原生 macOS 通知

### 基础层：TUI 通知路径处理审批和输入

权限请求和用户输入请求目前并不稳定地经过同一条 hook 路径。更现实的做法是启用 Codex TUI 对以下事件的通知：

- `approval-requested`
- `user-input-requested`

这已经能覆盖 CLI 使用中最关键的交互事件，特别是在终端本身不在前台的时候。

### Ghostty 扩展层：本地 app-server 桥接加标签页检测

Ghostty 暴露出了基础方案的一个缺口。Codex TUI 的通知语义是“终端失焦”，不是“当前 tab 失焦”。如果 Ghostty 还在前台，而你只是切到了同一窗口里的另一个 tab，那么默认 TUI 路径就太粗了。

为了解决这个问题，这个项目新增了 `codex-ghostty-notify` 和 `codex-ghostty-notify.mjs`。

这个包装器会做四件事：

1. 记录当前是从哪个 Ghostty tab 启动的 Codex 会话
2. 启动本地 `codex app-server`
3. 在 `codex --remote` 和这个 app-server 之间插入一个本地 WebSocket 代理
4. 直接从 app-server 协议里监听审批请求和用户输入请求

只要请求还处于挂起状态，这个包装器就会通过 AppleScript 轮询 Ghostty 当前选中的 tab；如果发现你已经不在启动 Codex 的那个 tab 上，它就会复用同一个 `codex-notify.mjs` 脚本发出 macOS 原生通知。

这样做的好处是：原生 `codex` 完全不需要改动，而更强的标签页感知能力则放在一个显式的包装命令后面。

## 为什么这样设计

这里的核心设计原则不是“越统一越好”，而是“尽量完整覆盖事件，同时尽量降低对现有工作流的侵入”。

- 如果只用官方 `notify` hook，那么任务完成可以通知，但权限请求会漏掉
- 如果只依赖 TUI 通知，那么任务完成路径较弱，而且 Ghostty 多标签场景下审批语义仍然不够精确
- 把小型 hook、内建 TUI 通知和可选的 Ghostty 专用桥接组合起来，才能在不改动原生 `codex` 的前提下得到最广的覆盖面

所以最终方案本质上是一个“覆盖优先、按需增强”的方案。

## 验证思路

这次使用了四层验证：

1. 脚本级验证
   直接向 `codex-notify.mjs` 输入一段模拟 JSON payload，确认脚本能正确解析并写日志
2. macOS 通知冒烟测试
   不禁用 `osascript`，直接调用脚本，确认桌面通知确实会弹出
3. 包装器语法和回退逻辑验证
   用 `node --check` 检查 `codex-ghostty-notify.mjs`，并确认非交互或不支持的调用会自动退回到原生 `codex`
4. 协议与集成验证
   从本地 Codex JSON schema 确认审批请求和用户输入请求确实会以 `app-server` server-request 事件形式暴露，再从 Ghostty 本地脚本字典确认当前选中 tab 的状态确实可以通过 AppleScript 查询

最后这一层很重要，因为当前这个沙箱环境对桌面自动化有限制，没法在这里完整跑一遍端到端的 Ghostty AppleScript 联调。所以这部分更多依赖本地协议定义和应用脚本字典来验证运行时设计是否成立。

## 这次过程中得到的经验

### 1. “让模型先问我要不要批准”不等于 CLI 的真实审批事件

这是最重要的一条经验。如果模型在对话里输出“是否批准我执行这个命令？”，那只是 assistant 文本，不是 Codex runtime 发出的真实审批事件。

测试时必须让 Codex 真正尝试执行一个需要批准的命令，只有这样才能触发真正的 `approval-requested`。

### 2. 事件语义比界面长得像不像更重要

两个界面看起来很像，并不代表它们来自同一条运行链路。对于通知插件来说，差别就在这里。我们必须挂在真实 runtime 事件上，而不是挂在一个“看起来像审批提示”的文本输出上。

### 3. 焦点语义同样很重要

“终端没有焦点”和“当前 tab 不是这个会话”是两种不同条件。Ghostty 把这件事暴露得很明显。一旦需求从“应用失焦提醒”变成“标签页失焦提醒”，单靠内建 TUI 通知就不够了。

### 4. 最稳的实现有时是分层实现

直觉上总会想找一个统一 hook 解决所有问题，但实际稳定可用的方案往往是分层的：

- 官方 `notify` hook 负责任务完成
- 内建 TUI 通知负责通用审批覆盖
- 只有在需要更细粒度焦点判断时，才引入 `app-server` 桥接

### 5. 即使是小工具，也应该保留日志能力

可选的 `CODEX_NOTIFY_LOG` 和 `CODEX_GHOSTTY_NOTIFY_LOG` 都很有价值。没有它们时，很难区分以下几种情况：

- 事件根本没发出来
- 事件发出来了，但桥接器或脚本解析失败
- 脚本执行成功了，但 macOS 没展示通知

## 下一步最值得做的改进

Ghostty 包装器已经解决了 Ghostty 标签页感知的问题，但更通用的下一步仍然是做一个“与终端无关”的焦点检测桥接层。那样可以继续沿用同一套 app-server 事件策略，同时把“焦点判断后端”替换成可适配不同终端应用的实现。

在这个基础上，后续还可以继续扩展到：

- Slack
- iPhone 推送
- 更丰富的交互式审批体验
