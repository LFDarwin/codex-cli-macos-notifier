# 实现说明

## 这次改动的目标

这个项目要解决的是 Codex CLI 在 macOS 上的一个很实际的问题：

- 当 Codex 在后台完成任务时，用户应该能及时收到提醒
- 当 Codex 需要命令批准或额外输入时，用户也应该被提醒，而不是错过交互时机

最终实现采用了混合方案，因为当前 Codex 把这些事件暴露在不同的机制里。

## 实现方法

### 官方 hook 路径

对于任务完成事件，最干净的方式是用官方 `notify` 配置。Codex 可以在一个 turn 完成后调用本地程序，并把 JSON payload 传给它。

在这个项目里，hook 指向的是 `codex-notify.mjs`。

它的职责刻意保持很小：

1. 接收原始 payload
2. 安全解析 JSON
3. 提取简洁的标题和正文
4. 可选追加一条机器可读的本地日志
5. 调用 `osascript` 发出原生 macOS 通知

### TUI 通知路径

权限请求和用户输入请求目前并不稳定地经过同一条 hook 路径。更现实的做法是启用 Codex TUI 对以下事件的通知：

- `approval-requested`
- `user-input-requested`

这已经能覆盖 CLI 使用中最关键的交互事件，特别是在终端不在前台的时候。

## 为什么这样设计

这里的核心设计原则不是“优雅统一”，而是“事件覆盖完整”。

- 如果只用官方 `notify` hook，那么任务完成可以通知，但权限请求会漏掉
- 如果只依赖终端通知，那么任务完成路径会比较弱，也不容易自定义
- 把两者组合起来，才能用最小代价获得最大的事件覆盖面

所以最终方案本质上是一个“覆盖优先”的方案，而不是“单机制优先”的方案。

## 验证思路

这次使用了三层验证：

1. 脚本级验证
   直接向 `codex-notify.mjs` 输入一段模拟 JSON payload，确认脚本能正确解析并写日志
2. macOS 通知冒烟测试
   不禁用 `osascript`，直接调用脚本，确认桌面通知确实会弹出
3. CLI 行为验证
   在真实交互式 Codex 中检查：
   - 完成事件是否走 hook 路径
   - 权限请求和输入请求是否在终端失焦时走 TUI 通知路径

## 这次过程中得到的经验

### 1. “让模型先问我要不要批准”不等于 CLI 的真实审批事件

这是最重要的一条经验。如果模型在对话里输出“是否批准我执行这个命令？”，那只是 assistant 文本，不是 Codex runtime 发出的真实审批事件。

测试时必须让 Codex 真正尝试执行一个需要批准的命令，只有这样才能触发真正的 `approval-requested`。

### 2. 事件语义比界面长得像不像更重要

两个界面看起来很像，并不代表它们来自同一条运行链路。对于通知插件来说，差别就在这里。我们必须挂在真实 runtime 事件上，而不是挂在一个“看起来像审批提示”的文本输出上。

### 3. 有时候最可靠的实现不是单一路径，而是组合路径

直觉上总会想找一个统一 hook 解决所有问题，但实际稳定可用的方案往往是把“最适合完成通知的机制”和“最适合审批通知的机制”组合起来。

### 4. 即使是小工具，也应该保留日志能力

可选的 `CODEX_NOTIFY_LOG` 很有价值。没有它时，很难区分以下几种情况：

- 事件根本没发出来
- 事件发出来了，但脚本解析失败
- 脚本执行成功了，但 macOS 没展示通知

## 下一步最值得做的改进

如果以后你希望审批通知也像完成通知一样强、一样可定制，那么下一步应该基于 Codex `app-server` 事件做一个本地桥接进程。那样完成事件和审批事件都可以统一走一条标准化通知管线，并且可以继续扩展到：

- Slack
- iPhone 推送
- 更丰富的交互式审批体验
